# Модель рисков продукта

Этот документ формализует знание о продукте, которое раньше существовало только в головах QA.
Агент использует эту модель для **классификации сбоев** и **принятия решений о релизе**.

---

## 1. Продукт

```
R7 Office Desktop Editors (Editors)
│
├── Общее (приложение целиком)
│   ├── Запуск приложения
│   ├── Стартовый экран
│   └── Навигация между редакторами
│
├── Документы
│   ├── Создание / открытие (.docx, .odt)
│   ├── Ввод и редактирование текста
│   ├── Форматирование
│   ├── Панель инструментов и вкладки
│   ├── Сохранение
│   └── Печать / предпросмотр
│
├── Таблицы
│   ├── Создание / открытие (.xlsx, .ods)
│   ├── Ввод данных в ячейки
│   ├── Формулы и вычисления
│   ├── Форматирование
│   ├── Сохранение
│   └── Печать / предпросмотр
│
└── Презентации
    ├── Создание / открытие (.pptx, .odp)
    ├── Добавление / редактирование слайдов
    ├── Вставка объектов (текст, изображения, фигуры)
    ├── Сохранение
    └── Показ слайдов
```

Editors — это одно приложение с тремя встроенными редакторами.
Стартовый экран, запуск и навигация — общие для всех трёх.

## 3. Карта рисков

### 3.1. Общее (приложение)

```yaml
area: Editors/Общее
risk_level: HIGH

critical_paths:
  - запуск приложения → стартовый экран отображается
  - переход из стартового экрана в каждый из редакторов
  - создание нового документа из стартового экрана

tolerances:
  ui_shift: допустимо (≤ 5px, элемент функционален)
  theme_diff: допустимо (различия в отображении темы между ОС)
  crash: НЕ допустимо
```

### 3.2. Документы

```yaml
area: Editors/Документы
risk_level: HIGH

critical_paths:
  - создание нового документа
  - открытие существующего (.docx, .odt)
  - ввод и редактирование текста
  - сохранение документа
  - печать / предпросмотр печати

additional_paths:
  - форматирование текста (шрифт, размер, жирный, курсив)
  - отмена / повтор действий
  - вставка объектов (таблицы, изображения)

smoke_critical_paths:
  - работа с вкладками панели инструментов (все вкладки должны переключаться)

tolerances:
  ui_shift: допустимо
  font_rendering_diff: допустимо (различия рендеринга шрифтов между ОС, текст читаем)
  toolbar_icon_mismatch: допустимо (незначительные отличия иконок)
  crash: НЕ допустимо
  data_loss: НЕ допустимо
  silent_corruption: НЕ допустимо (файл сохранён, но данные повреждены)
```

### 3.3. Таблицы

```yaml
area: Editors/Таблицы
risk_level: HIGH

critical_paths:
  - создание новой таблицы
  - открытие существующей (.xlsx, .ods)
  - ввод данных в ячейки
  - формулы и вычисления (результат должен совпадать с эталоном)
  - сохранение

additional_paths:
  - форматирование ячеек
  - сортировка и фильтрация
  - работа с листами (добавление, переключение, переименование)

tolerances:
  ui_shift: допустимо
  formula_precision_diff: НЕ допустимо (результат формулы должен совпадать)
  crash: НЕ допустимо
  data_loss: НЕ допустимо
```

### 3.4. Презентации

```yaml
area: Editors/Презентации
risk_level: MEDIUM

critical_paths:
  - создание новой презентации
  - открытие существующей (.pptx, .odp)
  - добавление / редактирование слайдов
  - сохранение

additional_paths:
  - вставка объектов (текст, изображения, фигуры)
  - показ слайдов
  - переходы между слайдами

tolerances:
  ui_shift: допустимо
  animation_diff: допустимо (различия анимаций/переходов между ОС)
  crash: НЕ допустимо
  data_loss: НЕ допустимо
```

## 4. Классификация сбоев

Каждый FAIL классифицируется по двум осям: **серьёзность** и **область**.

### Серьёзность (Severity)

| Уровень | Код | Описание | Влияние на решение |
|---|---|---|---|
| Критический | `CRITICAL` | Крэш, потеря данных, невозможность запуска | → **NO_GO** |
| Высокий | `HIGH` | Основная функция не работает (сохранение, печать, ввод) | → **NO_GO** |
| Средний | `MEDIUM` | Функция работает с ограничениями или обходным путём | → **GO_WITH_RISK** |
| Низкий | `LOW` | Косметические дефекты, незначительные UI-отклонения | → **GO** (с пометкой) |

### Правило Smoke-кейсов

**Все шаги smoke-кейсов классифицируются как severity `HIGH`.**

Smoke-набор проверяет базовую работоспособность продукта. Если хоть один шаг smoke-кейса не проходит — продукт не готов к дальнейшему тестированию и релизу. Это касается всех проверок в smoke, включая переключение вкладок панели инструментов: отсутствие вкладки — это не косметика, а неработающая функция.

### Область (Area)

| Область | Примеры |
|---|---|
| `DATA_INTEGRITY` | повреждение файла, потеря данных, неверные формулы |
| `CORE_FUNCTION` | открытие, сохранение, печать, ввод текста |
| `UI_LAYOUT` | смещение элементов, неверное отображение иконок |
| `PERFORMANCE` | медленный запуск, зависание при большом файле |
| `PLATFORM_SPECIFIC` | проблема воспроизводится только на одной ОС |

## 5. Допуски — что можно игнорировать осознанно

Система должна различать **реальный дефект** и **допустимое отклонение**:

| Отклонение | Допустимо? | Условие |
|---|---|---|
| Смещение UI-элементов | Да | ≤ 5px, элемент остаётся функциональным |
| Различие шрифтов между ОС | Да | текст читаем, layout не ломается |
| Отсутствие анимации | Да | функция работает без анимации |
| Различие в порядке пунктов меню | Да | все пункты присутствуют |
| Другой стиль скроллбара | Да | скролл работает |
| Крэш при любом сценарии | **Нет** | никогда не допустимо |
| Потеря данных | **Нет** | никогда не допустимо |
| Файл не открывается | **Нет** | файл валидный → должен открыться |
| Формула даёт неверный результат | **Нет** | точность вычислений обязательна |

## 6. Связь с тест-кейсами

Каждый тест-кейс привязывается к дереву продукта:

```yaml
case_id: 1
case_name: "Запуск редактора"
area: Editors/Общее
critical_path: true
severity_if_fail: CRITICAL
```

```yaml
case_id: 3
case_name: "Создание документа и проверка вкладок"
area: Editors/Документы
critical_path: true
severity_if_fail: HIGH
```

Это позволяет агенту:
- видеть, какие **области продукта** покрыты, а какие нет,
- автоматически определять вес каждого FAIL,
- формировать решение с привязкой к конкретному редактору,
- приоритизировать: сначала critical paths каждого редактора, потом additional.

## 7. Эволюция модели

Модель рисков — живой документ. Обновлять при:
- добавлении новой функциональности в редакторы,
- добавлении новой платформы,
- обнаружении нового типа дефекта,
- изменении приоритетов бизнеса (например, macOS стал HIGH),
- по результатам ретроспективы после инцидента.

Формат обновления: агент предлагает изменение → пользователь подтверждает → обновление фиксируется.

## 8. Политика предупреждений и fallback для smoke

### 8.1 UI drift без потери сигнала

- Совпадение ключевой проверки шага + обнаружение нового UI-элемента трактуется как `PASS` с warning.
- Базовое правило smoke: **новый элемент UI = warning (`UI_NEW_ELEMENT`)**, шаг не падает.
- Цель: не пропустить регрессию и одновременно рано подсветить необходимость обновления тестов/каталога.

### 8.2 UI-каталог как источник ожиданий

- Для Editors используется централизованный каталог секций и вкладок (например, `Главная`, `Вставка`, `Макет` и т.д.).
- Каталог содержит:
  - обязательные признаки (`required`) — отсутствие может вести к `TEST_FAIL`;
  - опциональные признаки (`optional`) — изменения фиксируются как warning.
- Поддержка smoke-кейсов выполняется через обновление каталога, а не точечные правки каждого кейса.

### 8.3 Fallback как риск покрытия

- Любой fallback (hotkey, OCR, coordinates, recursive file search, launch без debug) должен быть явно отражён в шаге:
  - `fallback_source`
  - `fallback_reason`
  - `warnings`
- Fallback не равен продуктовой ошибке, но снижает достоверность прогона и должен попадать в секции `Warnings`/`INFRA issues`.

### 8.4 Debug-launch policy

- На Windows запуск выполняется цепочкой:
1. `debug` (с `--ascdesktop-support-debug-info`);
2. при неуспехе — `standard` (без debug-флага).
- Если debug-порт недоступен, тест не должен падать сразу: сохраняется fallback-цепочка действий.
- Если оба launch-режима неуспешны, это инфраструктурный риск (`INFRA_FAIL`/`BLOCKED`) с обязательной фиксацией в отчёте.

## 9. Target model alignment для Editors

Для Editors целевая модель верификации фиксируется в
`docs/methodology/TARGET_MODEL_GLOSSARY.md` и применяется совместно с этим
`RISK_MODEL.md`.

Обязательные правила применения:

- `risk-based decisioning` всегда опирается на критические пути и допуски текущего
  документа; glossary не переопределяет риск-политику, а стандартизирует термины.
- Для critical path проверки должны подтверждаться approved verification sources из
  glossary с приоритетом semantic/assertion и file-model сигналов.
- `geometry-aware` признаки трактуются только в рамках tolerance этой модели
  (например, `ui_shift <= 5px` допустим при сохраненной функциональности).
- Любой fallback отражается как риск достоверности сигнала и влияет на
  `run_confidence` через `DECISION_ENGINE.md`.

Definition of Done для миграции:
- применяется из `TARGET_MODEL_GLOSSARY.md`, раздел 3;
- миграция не считается завершенной, если хотя бы один пункт DoD не выполнен.
