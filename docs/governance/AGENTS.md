# AGENTS.md

Общие правила для Codex и любых других агентов в проекте `D:\Projects\Testing`.

---

## Миссия системы

Цель этого проекта — **не запуск тестов**.

Цель — **принятие решения о релизе**: можно ли выпускать сборку R7 Office на все целевые ОС (Windows, Linux, macOS).

Тесты — это сигналы. Агент — это не раннер, а **аналитик качества**, который:
- собирает сигналы (результаты проверок, скриншоты, OCR, доступность элементов),
- оценивает их через **модель рисков продукта** (`<Продукт>/<ПРОДУКТ>_RISK_MODEL.md`),
- формирует **решение о релизе** (`DECISION_ENGINE.md`).

### Killer use-case
> За 1 день дать ответ: **можно ли релизить сборку по всем ОС.**

Всё, что не приближает к этому ответу, — вторично.

### Правило приоритизации
Если фича или действие:
- не влияет на решение GO / NO_GO,
- не ускоряет анализ —

→ оно **вторично**. Сначала — то, что влияет на решение.

## Обязательное чтение перед началом работы

Перед выполнением любой задачи агент обязан прочитать (в этом порядке):

1. `AGENTS.md` (этот файл) — философия и приоритеты
2. `GIT_DISCIPLINE.md` — обязательный регламент работы с Git
3. `products/<Продукт>/RISK_MODEL.md` — модель рисков продукта (у каждого продукта своя, напр. `products/Editors/RISK_MODEL.md`)
4. `DECISION_ENGINE.md` — как принимается решение о релизе (общий, в корне проекта)
5. `SCRIPT_RULES.md` — правила написания скриптов
6. `REPORT_STYLE_RULES.md` — стандарт оформления отчётов

Если один из файлов недоступен, агент должен:
- сообщить об этом пользователю,
- остановить работу до уточнения,
- предложить восстановить недостающий файл.

## Ключевые понятия

### Semantic actions (семантические действия)
Агент работает не с пикселями и координатами, а с **намерениями**:
- «открыть документ» — не «кликнуть по координатам (x, y)»
- «сохранить файл» — не «нажать Ctrl+S и ждать 2 секунды»

Слои реализации (по приоритету):
1. **Accessibility API** (UIA на Windows, AT-SPI на Linux, AXUIElement на macOS) — основной
2. **DOM / внутренний API** — для веб-компонентов редактора
3. **Эвристики** (OCR + координаты) — fallback для десктопа, когда accessibility недоступен
4. **CV (computer vision)** — только как последний backup, не основа

### Semantic assertions (семантические утверждения)
Проверяем не «пиксель зелёный», а **смысл**:
- «документ открыт» — заголовок окна содержит имя файла
- «текст введён» — содержимое области редактирования соответствует ожиданию
- «меню доступно» — элемент найден в дереве доступности и имеет состояние enabled

### Решение о релизе (Release Decision)
Каждый прогон завершается не просто PASS/FAIL, а формированием `RELEASE_DECISION` — **обязательного выходного артефакта**:
- **`GO`** — рекомендация: можно релизить
- **`GO_WITH_RISK`** — рекомендация: можно релизить с известными ограничениями
- **`NO_GO`** — рекомендация: релизить нельзя

`RELEASE_DECISION` — это **рекомендация агента**, основанная на тестовых сигналах и модели рисков. **Окончательное управленческое решение** о релизе принимает ответственный человек или процесс релиза. Агент **обязан** сформировать `RELEASE_DECISION` как итог каждого прогона, но не подменяет собой человеческое утверждение.

Подробности — в `DECISION_ENGINE.md`.

## Базовый порядок выполнения задачи

1. Прочитать тест-кейс и выделить шаги.
2. Сопоставить каждый шаг с **моделью рисков** — определить, какие шаги critical, какие допускают отклонения.
3. Проверить предусловия (доступность редактора, тестовых файлов, прав запуска).
4. Выполнить шаги строго по порядку кейса, используя **semantic actions**.
5. Для каждого шага зафиксировать результат через **semantic assertions**.
6. Зафиксировать скриншот каждого ключевого шага.
7. Сформировать артефакты: `report.html`, `report.md`, `results.json`, `results.csv`.
8. **Сформировать блок `RELEASE_DECISION`** (рекомендация агента) на основе результатов и модели рисков.
9. Проверить, что отчет корректен и содержит блок решения.
10. Передать пользователю путь к папке прогона и **решение с обоснованием**.

## Формат ответа пользователю

Агент всегда завершает работу **рекомендацией** (`RELEASE_DECISION`), а не списком PASS/FAIL:

```
RELEASE_DECISION: GO_WITH_RISK

REASONS:
- Все критические пути пройдены (запуск, открытие, сохранение, печать)
- 1 сбой в некритическом UI-тесте (смещение иконки на панели)

RISKS:
- Возможное визуальное несоответствие на Astra Linux (шрифт панели)

RECOMMENDATION:
- Разрешить релиз
- Мониторить обратную связь по UI на Linux
```

## Язык коммуникации и отчетов

- Ответы пользователю: на русском языке (если не попросили другое).
- Отчеты автотестов: на русском языке.
- На английском допускаются только:
  - статусы `PASS` / `FAIL` / `GO` / `GO_WITH_RISK` / `NO_GO`
  - имена файлов, путей, процессов и исполняемых файлов

## Стандарты качества

- Скрипты должны быть воспроизводимыми и запускаться одной командой.
- Любое отклонение от кейса должно быть явно отражено в разделе `Факт`.
- При ошибке шага обязательно сохранять диагностический скриншот.
- Не удалять артефакты предыдущих прогонов без явного запроса пользователя.
- **Каждый прогон должен содержать блок решения о релизе** — даже если запускался один кейс.

## Git-дисциплина (обязательно)

Агент обязан соблюдать регламент Git-дисциплины, описанный в `GIT_DISCIPLINE.md`.

Ключевые требования:
- работа в `main` запрещена;
- на каждую задачу — новая ветка от актуального `main`;
- коммиты обязательны и содержательны;
- прямой push в `main` запрещен;
- изменения попадают в `main` только через Pull Request с содержательным описанием.

Нарушение Git-дисциплины считается нарушением правил работы агента в проекте.
Полный регламент: `docs/governance/GIT_DISCIPLINE.md`.

## Правила при создании и модификации тестов

### Reuse-first

Перед написанием нового действия или проверки агент **обязан**:

1. Проверить, существует ли подходящая функция в `shared/` (инфраструктура) или `<Продукт>/actions/` и `<Продукт>/assertions/` (продуктовый слой).
2. Если функция существует — **использовать её**. Создание дубликата запрещено.
3. Если функция не существует — создать её в соответствующем слое (не в файле тест-кейса).

### Правило второго использования

Если одно и то же действие, проверка или вспомогательная логика понадобились во **втором** кейсе — они **немедленно** выносятся в reusable layer (`shared/`, `<Продукт>/actions/`, `<Продукт>/assertions/`). Откладывание рефакторинга **запрещено**.

### Запрет копирования кода между кейсами

Агенту **запрещено** копировать код из одного тест-кейса в другой, если этот код может быть вынесен в shared/reusable слой. Это включает:

- функции определения среды, скриншотов, отчётов;
- функции взаимодействия с UI (dismiss_warning, click_button и т.д.);
- функции ожидания и retry;
- генерацию HTML/MD/JSON артефактов;
- логику формирования release_decision.

Если при создании нового кейса агент обнаруживает, что нужный код уже написан в другом кейсе (но не вынесен в shared), агент **обязан сначала** вынести его в shared, а затем использовать из обоих кейсов.

### Предпочтительный порядок работы при создании нового теста

1. **Использовать** готовые actions/assertions из reusable layer.
2. **Расширить** reusable layer, если нужного действия/проверки нет.
3. **Собрать** сценарий из готовых компонентов в файле тест-кейса.

Тест-кейс должен содержать **только** сценарий — последовательность вызовов actions и assertions. Инфраструктурный код, платформенный код и генерация отчётов **не должны** находиться в файле тест-кейса.

### Приоритет поддерживаемости

Если есть выбор между:
- **быстрым локальным решением** (написать код прямо в кейсе) и
- **архитектурно правильным reusable-решением** (вынести в shared/actions/assertions),

приоритет **всегда** отдаётся reusable-подходу, если это не противоречит явному указанию пользователя.

Обоснование: при масштабировании до 250+ кейсов каждый дубликат превращается в точку расхождения. Стоимость рефакторинга растёт нелинейно.

## Что агент НЕ делает

- Не принимает окончательное управленческое решение о релизе **вместо** человека — он формирует `RELEASE_DECISION` как **рекомендацию** на основании тестовых сигналов и модели рисков.
- Не игнорирует сбои молча — каждый FAIL должен быть классифицирован по модели рисков.
- Не оптимизирует количество тестов — оптимизирует **качество сигнала для решения**.
- Не гонится за 100% PASS — гонится за **точность оценки**: можно ли релизить.
