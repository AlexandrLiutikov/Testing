# GIT_DISCIPLINE.md

Обязательный регламент работы AI-агентов с Git в данном репозитории.

Этот документ — не рекомендация. Это обязательный operational rule set.
Нарушение любого правила означает, что задача выполнена некорректно.

---

## 1. Назначение

Данный документ регулирует поведение AI-агентов при работе с Git в репозитории проекта.
Документ обязателен для всех AI-агентов, выполняющих любые изменения в репозитории.

---

## 2. Область действия

Правила распространяются на все виды агентской работы, приводящие к изменениям в репозитории:

- изменение кода (скрипты, модули, конфигурации);
- изменение тестов и тест-сценариев;
- изменение структуры проекта (каталоги, пути, точки запуска);
- изменение документации;
- рефакторинг;
- подготовка PR.

Если агент изменяет хотя бы один файл в репозитории — данный регламент действует полностью.

---

## 3. Обязательная модель работы с Git

Агент обязан работать **только** через отдельную рабочую ветку.

- Работа в `main` запрещена.
- Любые изменения попадают в `main` только через Pull Request.
- PR обязателен для каждой задачи.

Исключений нет.

---

## 4. Правила создания веток

### 4.1. Одна задача — одна ветка

- На каждую задачу агент обязан создать **новую** рабочую ветку.
- В одной ветке запрещено вести несколько независимых задач.
- Запрещено переиспользовать старую рабочую ветку для новой задачи.
- Запрещено продолжать новую задачу в ветке от предыдущей задачи, если это не оговорено пользователем явно.

### 4.2. Ветка создается от актуального main

Перед созданием ветки агент обязан:

1. Переключиться на `main`.
2. Выполнить `git pull` для получения актуального состояния.
3. Убедиться, что новая ветка создается от свежего `main`.

Создание ветки от устаревшего `main` или от другой рабочей ветки (без явного указания пользователя) запрещено.

---

## 5. Правила именования веток

Агент обязан использовать следующий шаблон:

```
<тип>/<идентификатор-задачи>-<краткое-описание>
```

Допустимые типы:

| Тип | Назначение |
|---|---|
| `feature` | Новая функциональность |
| `fix` | Исправление бага |
| `refactor` | Рефакторинг без изменения поведения |
| `chore` | Служебные изменения (CI, зависимости, конфигурация) |
| `docs` | Изменения только в документации |
| `test` | Добавление или изменение тестов |

Примеры:
- `feature/EDIT-42-smoke-test-save-docx`
- `fix/EDIT-15-screenshot-path-linux`
- `docs/GOV-3-git-discipline`

Если идентификатор задачи отсутствует, допускается краткое описание без идентификатора:
- `fix/ocr-timeout-retry`

Запрещены имена веток без смысловой нагрузки: `test`, `tmp`, `branch1`, `dev`, `my-branch`.

---

## 6. Правила коммитов

### 6.1. Коммиты обязательны

Агент обязан фиксировать изменения осмысленными коммитами по ходу выполнения задачи.
Запрещено оставлять всю работу как один незакоммиченный набор изменений.

### 6.2. Коммиты должны отражать логические этапы

Каждый коммит должен соответствовать одному логическому этапу работы:
- добавление нового модуля;
- исправление конкретной проблемы;
- обновление документации;
- рефакторинг отдельного компонента.

Запрещено делать один гигантский коммит на всю задачу, если задача содержит несколько логически раздельных изменений.

### 6.3. Сообщения коммитов должны быть содержательными

Commit message обязан описывать, **что именно** изменено.

Запрещены сообщения:
- `fix`
- `update`
- `changes`
- `tmp`
- `wip`
- `123`
- `test`
- `.`
- пустые сообщения

Допустимые примеры:
- `Add smoke test for document save in DOCX format`
- `Fix screenshot capture path on Linux`
- `Refactor report generation to use shared template`
- `Update RISK_MODEL.md with new critical paths`

### 6.4. Запрет на коммит чувствительных данных

Агенту запрещено коммитить:
- файлы с секретами, токенами, паролями (`.env`, `credentials.json` и т.п.);
- файлы виртуального окружения (`.venv/`);
- артефакты сборки, кеши IDE, временные файлы ОС.

Если `.gitignore` не покрывает такие файлы — агент обязан обновить `.gitignore` до коммита.

---

## 7. Правила Pull Request

### 7.1. PR обязателен

Любые изменения попадают в `main` только через Pull Request.
Агенту запрещено обходить PR-процесс.

### 7.2. PR создается из рабочей ветки

PR создается только из рабочей ветки задачи в `main`.
Запрещено создавать PR из `main` в `main`.

### 7.3. Содержательное описание PR обязательно

В описании PR агент обязан указать:

1. **Что сделано** — краткое описание изменений.
2. **В рамках какой задачи** — идентификатор или описание задачи.
3. **Какие части проекта затронуты** — модули, каталоги, слои.
4. **Что было проверено** — какие проверки выполнены перед PR.
5. **Риски, ограничения, допущения** — если есть.

Если изменения затрагивают:
- архитектуру проекта,
- структуру каталогов,
- пути к ресурсам или точки запуска,
- контракты между модулями,
- сценарии запуска,

— это должно быть **отдельно** указано в описании PR.

Пустое, формальное или автоматически сгенерированное без адаптации описание PR недопустимо.

### 7.4. Размер PR

PR должен быть обозримым. Если изменения крупные — это обусловлено задачей, и размер должен быть явно указан в описании PR с обоснованием.

---

## 8. Самопроверка агента перед PR

Перед созданием PR агент обязан выполнить самопроверку:

- [ ] Ветка соответствует одной задаче.
- [ ] Нет файлов, не относящихся к задаче.
- [ ] Нет временных debug-артефактов (print/console.log/временные логи/закомментированный код).
- [ ] Нет случайных изменений вне задачи.
- [ ] Если изменения затрагивают процесс, архитектуру, структуру каталогов, пути или сценарии запуска — связанная документация обновлена.
- [ ] Если в проекте доступны линтеры, тесты или другие проверки — они выполнены.
- [ ] `.gitignore` покрывает артефакты, которые не должны попадать в репозиторий.
- [ ] Commit history чистая: нет мусорных коммитов, сообщения содержательные.

---

## 9. Работа с конфликтами

- Агент обязан разрешать конфликты **осознанно**, анализируя обе стороны.
- Запрещено механически принимать одну сторону (`--theirs` / `--ours`) без анализа.
- После разрешения конфликтов агент обязан повторно проверить корректность результата.
- Если конфликт затрагивает критическую логику и агент не уверен в правильном разрешении — агент обязан остановиться и запросить решение у пользователя.

---

## 10. Ограничение на смешивание изменений

- Запрещено смешивать в одном PR разные независимые задачи.
- Запрещено смешивать функциональные изменения и побочный рефакторинг без явной необходимости.
- Запрещено делать «заодно» массовые несвязанные изменения.

Одна задача — один PR. Одна цель — один PR.

---

## 11. Debug-cleanup и documentation-sync

### 11.1. Debug-cleanup

Перед финальным PR агент обязан удалить:
- временные логи и print-отладку;
- закомментированный код, оставленный для отладки;
- временные тестовые данные;
- TODO-комментарии, относящиеся к текущей задаче и уже решенные.

Исключение: если задача явно требует сохранения отладочных инструментов.

### 11.2. Documentation-sync

Если изменения затрагивают:
- процесс работы,
- архитектуру,
- governance-правила,
- структуру каталогов,
- сценарии запуска,
- зависимости,

— агент обязан обновить связанную документацию в том же PR.

Запрещено менять поведение системы без обновления документации, которая это поведение описывает.

### 11.3. Path/structure awareness

Если агент переносит файлы, меняет каталоги, импорты, точки запуска или пути к ресурсам — он обязан:
- проверить, что все ссылки на перемещенные объекты обновлены;
- явно отразить структурные изменения в описании PR.

---

## 12. Запрещенные действия

Агенту **запрещено**:

| # | Действие |
|---|---|
| 1 | Работать в `main` (изменения, коммиты, разработка) |
| 2 | Коммитить в `main` |
| 3 | Выполнять `git push` в `main` |
| 4 | Выполнять `git push --force` в любую ветку без явного указания пользователя |
| 5 | Вести несколько задач в одной ветке |
| 6 | Переиспользовать старую task-branch для новой задачи |
| 7 | Создавать ветку от устаревшего `main` |
| 8 | Оставлять пустые или бессодержательные commit messages |
| 9 | Создавать PR без содержательного описания |
| 10 | Смешивать несвязанные изменения в одном PR |
| 11 | Оставлять debug-мусор в финальном PR |
| 12 | Коммитить чувствительные данные (секреты, токены, пароли) |
| 13 | Менять поведение системы без обновления документации |
| 14 | Механически разрешать конфликты без анализа |
| 15 | Выполнять `git reset --hard`, `git clean -f` и другие деструктивные операции без явного указания пользователя |
| 16 | Завершать задачу с незакоммиченными рабочими изменениями |

---

## 13. Обязательный рабочий сценарий

Агент обязан выполнять работу в следующей последовательности:

```
1. git checkout main
2. git pull origin main
3. git checkout -b <тип>/<описание>       ← новая ветка от актуального main
4. Выполнить изменения по задаче
5. Делать осмысленные коммиты по ходу работы
6. Выполнить самопроверку (раздел 8)
7. git push -u origin <имя-ветки>
8. Создать Pull Request в main
9. Оформить содержательное описание PR (раздел 7.3)
10. Завершить работу
```

Отклонение от этой последовательности допускается только при явном указании пользователя.

---

## 14. Условия, при которых задача не считается завершенной

Агент **не имеет права** считать задачу завершенной, если:

- работа выполнена в `main`;
- не создана отдельная рабочая ветка;
- ветка создана не от актуального `main` (без явного указания);
- изменения не зафиксированы осмысленными коммитами;
- commit messages бессодержательны;
- не подготовлен Pull Request;
- PR не имеет содержательного описания;
- в PR остались debug-артефакты, мусор или несвязанные изменения;
- затронутая документация не обновлена;
- нарушено любое правило данного регламента.

При обнаружении нарушения агент обязан исправить его до завершения задачи.
