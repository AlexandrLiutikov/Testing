# Правила написания автотест-скриптов

Документ фиксирует единый подход к созданию и поддержке скриптов в проекте.
Язык скриптов: **Python**. Все тесты пишутся на Python и запускаются на всех целевых платформах.

---

## 1. Общие требования
- Скрипты должны быть воспроизводимыми: запуск одного файла должен давать полный результат без ручных действий.
- Каждый скрипт принимает параметры через аргументы командной строки или конфигурационный файл: путь к редактору (`--editor-path`) и путь для артефактов (`--output-dir`).
- Перед запуском теста скрипт должен закрывать ранее открытые процессы редактора (`editors`, `editors_helper`), если предусловия кейса не требуют иного.
- Завершение редактора после теста определяется стратегией запуска (см. раздел 10.1).
- **Скрипты должны работать на всех целевых платформах** (см. `<Продукт>/<ПРОДУКТ>_RISK_MODEL.md`, раздел «Проверяемые пакеты»). Платформо-зависимый код выносится в отдельный слой (см. раздел 12).

## 2. Именование
- Имя скрипта должно соответствовать тест-кейсу.
- Рекомендуемый формат: `N_краткое_название_кейса.py`.
- Для единообразия отчета имя кейса в мета-блоке должно совпадать с названием тест-кейса.

## 3. Структура файла тест-кейса

Файл тест-кейса содержит **только сценарий** (см. раздел 13 — Архитектура слоёв).

### 3.1 Что ДОЛЖНО быть в файле тест-кейса

- **`CASE_META`** — метаданные кейса (id, name, area, risk_level, critical_path).
- **Импорты** из reusable слоёв: `actions/`, `assertions/`, `shared/`.
- **Сценарий** — последовательность вызовов semantic actions и semantic assertions в порядке шагов тест-кейса.
- **`main()`** — orchestration-level функция: парсинг аргументов, вызов сценария, формирование результата. Обязательно наличие `try/except/finally`.
- Для каждого шага формируется запись по контракту `step_result` (см. раздел 15).

### 3.2 Что НЕ ДОЛЖНО быть в файле тест-кейса

- Платформенные реализации (`subprocess`, `ctypes`, `pywinauto`, `pyatspi` и т.д.).
- Определения shared utility-функций (скриншоты, отчёты, среда, decision).
- Генерация отчётов (`generate_html`, `generate_md`, `generate_csv`).
- Низкоуровневая работа с ОС (прямой доступ к реестру, файловой системе ОС, дереву доступности).
- Локальные `wait_until` / `retry` helpers — они должны быть в Infrastructure Layer (см. раздел 18).

## 4. Логирование и статусы
- Каждый шаг получает `status`: `PASS`, `FAIL` или `BLOCKED` (полный контракт — раздел 15).
- `PASS` не означает «без замечаний»: шаг может иметь `warnings` и/или `fallback_source` без перевода в `FAIL`.
- При `status = FAIL` обязательно указывается `failure_type`: `TEST_FAIL` (дефект продукта) или `INFRA_FAIL` (сбой среды/инструмента).
- При `status = BLOCKED` шаг оформляется по правилам раздела 20 (`failure_type = BLOCKED`, `failure_severity = null`, `failure_area = null`).
- Для каждого шага фиксировать краткий и проверяемый факт выполнения.
- При ошибке в `except` обязательно добавлять отдельную запись об ошибке с причиной и корректным `failure_type`.

## 5. Скриншоты и артефакты
- Скриншот обязателен для каждого ключевого шага.
- Все артефакты сохранять в отдельную папку прогона: `products/<Продукт>/artifacts/<run_id>/`.
- Минимальный набор файлов:
  - `report.html`
  - `report.md`
  - `results.json`
  - `results.csv`
  - PNG-скриншоты шагов

## 6. Требования к отчету (обязательно)
- Отчет формировать по принятому шаблону проекта (`REPORT_STYLE_RULES.md`).
- **Вся текстовая информация в отчете должна быть на русском языке.**
- Это требование распространяется на:
  - названия шагов (`Шаг`)
  - ожидаемые результаты (`Ожидание`)
  - фактические результаты (`Факт`)
  - служебные пояснения в meta-блоке и подписях
- Допускаются технические обозначения на английском только для:
  - статусов (`PASS`/`FAIL`/`BLOCKED`/`GO`/`GO_WITH_RISK`/`NO_GO`)
  - имен файлов и путей
  - названий исполняемых файлов/процессов

## 7. Кодировка
- Скрипты и отчеты сохранять в UTF-8.
- Не допускать «кракозябр»; при проблемах с отображением использовать безопасное экранирование HTML.

## 8. Проверка перед сдачей
- Скрипт запускается командой `python <script.py>` без ошибок на всех целевых ОС.
- Отчет открывается локально в браузере и корректно отображает русский текст.
- Количество шагов в отчете соответствует тест-кейсу.
- Статусы и счетчики `PASS/FAIL` совпадают с фактическими результатами шагов.
- **Блок решения о релизе присутствует** (см. раздел 11).
- **Мета-блок содержит информацию о среде** (ОС, пакет, разрешение экрана — см. раздел 12).

## 9. Формулировки в отчете (устойчивость)
- Не указывать в Факт точные численные значения, чувствительные к среде (координаты, пиксели).
- Разрешено указывать качественное изменение: яркость фона снизилась, пункт меню подсвечен, модальное окно появилось/закрыто.
- Внутренние технические значения не должны попадать в пользовательский отчет.

## 10. Дополнительные правила

### 10.1 Завершение редактора и цепочка кейсов

Для каждого тест-кейса **обязательно** явно задавать предусловия и постусловия.

Стратегия завершения приложения:

- **Standalone-кейс** (запускается отдельно): по умолчанию **закрывает** редактор после выполнения. Это поведение по умолчанию.
- **Кейс в цепочке** (постусловие требует оставить приложение открытым для следующего кейса): редактор **не закрывается**. Следующий кейс наследует состояние приложения.
- **Последний кейс цепочки**: **закрывает** редактор.

Стратегию определяют предусловия/постусловия кейса или run-level orchestration (`run_all`). Если кейс запускается как standalone, а его постусловия не указывают иного — редактор закрывается.

### 10.2 Проверка вместо формального PASS
- Статус PASS ставится только после подтверждения ожидаемого результата.
- Для каждого шага должен быть проверяемый критерий (видимый элемент, состояние вкладки, текст, модальное окно и т.д.).
- Запрещено выставлять PASS сразу после действия без фактической проверки результата.
- Если действие выполнено, но ожидаемое состояние не подтверждено → FAIL.

### 10.3 Приоритет способа взаимодействия с UI (semantic actions)
Скрипт должен взаимодействовать с приложением через **смысловые действия**, а не координаты:

**Приоритет реализации:**
1. **Accessibility API** — поиск элемента по имени/роли через дерево доступности (UIA на Windows, AT-SPI на Linux, AXUIElement на macOS)
2. **Идентификация по тексту** — OCR + поиск текстового паттерна
3. **Координатный клик** — только как последний fallback, с комментарием почему

Скрипт должен быть написан так, чтобы замена слоя взаимодействия (например, UIA → AT-SPI) не требовала переписывания логики теста.

**Правило соответствия шагу тест-кейса (обязательно):**
- Если в шаге тест-кейса явно указано действие **через кнопку/элемент интерфейса** (например: «кликнуть кнопку "Сохранить"»), действие в автоматизации должно выполняться через этот UI-элемент.
- **Запрещено** заменять такой шаг горячей клавишей (`Ctrl+S`, `Ctrl+Z`, `Ctrl+Y` и т.п.) как основной путь выполнения.
- Горячая клавиша допускается только как **fallback**, если основной UI-путь недоступен по объективной причине (ограничение Accessibility/DOM/OCR), и это явно зафиксировано в комментарии и логе шага.

### 10.4 Semantic assertions (семантические проверки)
Каждая проверка должна отвечать на вопрос **«что произошло с точки зрения пользователя»**, а не «что вернул API»:

| Правильно (семантическая) | Неправильно (техническая) |
|---|---|
| Документ открыт — заголовок окна содержит имя файла | FindWindow вернул handle != 0 |
| Текст введён в область редактирования | SendKeys отработал без исключения |
| Вкладка «Вставка» активна и подсвечена | Яркость пикселя (x,y) > 200 |
| Меню «Файл» открыто, видны пункты | OCR нашёл 3 из 5 токенов |

### 10.5 Обязательные ожидания загрузки
- После ключевых действий — явное ожидание, заданное тест-кейсом.
- Фиксация результата шага выполняется только после завершения загрузки.

### 10.6 Обработка модальных окон и плашек
- Обязательно предусматривать обработку всплывающих окон: основной путь через кнопку (`OK`), резервный через клавиатуру.
- Появление и закрытие модального окна фиксировать как отдельные проверяемые действия.

### 10.7 Требования к отчету при ошибках
- Для `FAIL` обязательно указывать причину и прикладывать скриншот.
- В поле `Факт` запрещено указывать технический способ проверки. Только наблюдаемый пользовательский результат.

### 10.8 Кодировка и воспроизводимость
- Скрипты сохранять в UTF-8.
- Один запуск скрипта должен воспроизводимо формировать полный набор артефактов.

### 10.9 Единый формат артефактов
- Имена скриптов и папок прогонов — по шаблону проекта.
- Обязательные артефакты: `report.html`, `report.md`, `results.json`, `results.csv`, скриншоты.

### 10.10 Устойчивость OCR-валидации
- Поиск Tesseract через `shutil.which('tesseract')` с fallback на стандартные пути установки по ОС.
- Не менее двух режимов OCR (--psm 6, --psm 11) с объединением результата.
- Мягкое совпадение токенов (допуск OCR-искажений).
- FAIL по OCR — только после исчерпания всех устойчивых проверок.

### 10.11 Контроль перед сдачей
- Проверять синтаксис скрипта и соответствие шагов тест-кейсу.
- Отчет должен содержать корректные счетчики и соответствовать фактическим результатам.

## 11. Классификация сбоев и решение о релизе

**Каждый скрипт обязан формировать блок решения о релизе.**

### 11.1 Метаданные кейса

Каждый скрипт должен содержать метаданные, связывающие его с моделью рисков (`<Продукт>/<ПРОДУКТ>_RISK_MODEL.md`):

```python
CASE_META = {
    "case_id": 1,
    "case_name": "Запуск редактора",
    "area": "Editors/Общее",
    "risk_level": "HIGH",
    "critical_path": True,
}
```

### 11.2 Классификация каждого FAIL

При сбое шага скрипт должен классифицировать его:

```python
step_result["failure_severity"] = "MEDIUM"    # CRITICAL | HIGH | MEDIUM | LOW
step_result["failure_area"]     = "UI_LAYOUT"  # DATA_INTEGRITY | CORE_FUNCTION | UI_LAYOUT | PERFORMANCE | PLATFORM_SPECIFIC
step_result["failure_detail"]   = "Смещение иконки вкладки на панели инструментов"
```

### 11.3 Формирование решения

После выполнения всех шагов скрипт формирует блок решения по алгоритму из `DECISION_ENGINE.md`:

```python
decision = {
    "verdict": "GO_WITH_RISK",
    "reasons": ["Все критические пути пройдены", "1 сбой в UI с severity MEDIUM"],
    "risks": ["Смещение иконки вкладки «Вставка»"],
    "recommendations": ["Разрешить релиз", "Создать задачу на исправление UI"],
    "stats": {"total": 5, "passed": 4, "failed": 1, "critical_path_coverage": "3/3"},
}
```

Блок `release_decision` записывается в `results.json` и отображается в `report.html`.

### 11.4 Сводный прогон (run_all)

При запуске нескольких кейсов через `run_all.py`:
- Каждый кейс формирует своё локальное решение.
- `run_all.py` агрегирует решения: сводный вердикт = наихудший из кейсов.
- Сводный отчет содержит раздел «Решение о релизе» с полной аргументацией.

## 12. Кроссплатформенность и определение среды

Тесты прогоняются на всех ОС, на которых разворачивается редактор. Скрипт обязан автоматически определять среду выполнения и фиксировать её в отчёте.

### 12.1 Обязательный сбор информации о среде

При старте каждого прогона скрипт собирает:

```yaml
environment:
  os_family: Linux | Windows | macOS
  os_name: "Astra Linux 1.7.5"        # конкретная ОС и версия
  os_version: "1.7.5"
  architecture: x86_64 | x86 | arm64
  package: "astra-signed.deb"          # какой пакет установлен
  screen_resolution: "1920x1080"       # текущее разрешение экрана
  display_scale: "100%"                # масштаб (если определяется)
  editor_version: "2024.1.234"         # версия установленного редактора
  editor_path: "/opt/r7-office/..."    # путь к исполняемому файлу
  hostname: "test-astra-01"            # имя машины (для идентификации стенда)
  run_timestamp: "2026-04-04 14:30:00"
```

### 12.2 Определение ОС

Скрипт определяет ОС автоматически, без ручного ввода:

```python
import platform

def detect_os():
    system = platform.system()  # 'Linux', 'Windows', 'Darwin'

    if system == 'Linux':
        # /etc/os-release — стандарт для всех целевых дистрибутивов
        info = {}
        with open('/etc/os-release') as f:
            for line in f:
                k, _, v = line.strip().partition('=')
                info[k] = v.strip('"')
        return {
            'os_family': 'Linux',
            'os_name': info.get('PRETTY_NAME', 'Linux'),
            'os_version': info.get('VERSION_ID', ''),
        }

    elif system == 'Windows':
        ver = platform.version()        # '10.0.22631'
        release = platform.release()    # '10', '11'
        return {
            'os_family': 'Windows',
            'os_name': f'Windows {release}',
            'os_version': ver,
        }

    elif system == 'Darwin':
        ver = platform.mac_ver()[0]     # '14.2.1'
        return {
            'os_family': 'macOS',
            'os_name': f'macOS {ver}',
            'os_version': ver,
        }
```

### 12.3 Определение пакета

Скрипт определяет, какой именно пакет установлен:

```python
import subprocess

def detect_package():
    system = platform.system()

    if system == 'Linux':
        # DEB-based (Debian, Ubuntu, Astra)
        result = subprocess.run(
            ['dpkg-query', '-W', '-f', '${Package} ${Version} ${Architecture}',
             'r7-office-editors'],
            capture_output=True, text=True
        )
        if result.returncode == 0:
            return {'manager': 'deb', 'info': result.stdout.strip()}

        # RPM-based (ALT, RedOS, Rosa)
        result = subprocess.run(
            ['rpm', '-q', '--qf', '%{NAME} %{VERSION} %{ARCH}',
             'r7-office-editors'],
            capture_output=True, text=True
        )
        if result.returncode == 0:
            return {'manager': 'rpm', 'info': result.stdout.strip()}

    elif system == 'Windows':
        # реестр: winreg
        import winreg
        ...

    elif system == 'Darwin':
        # /Applications/R7-Office.app/Contents/Info.plist
        import plistlib
        ...
```

### 12.4 Определение разрешения экрана

```python
def detect_screen():
    system = platform.system()

    if system == 'Linux':
        result = subprocess.run(
            ['xrandr', '--current'], capture_output=True, text=True
        )
        for line in result.stdout.splitlines():
            if '*' in line:
                return line.split()[0]  # '1920x1080'

    elif system == 'Windows':
        import ctypes
        user32 = ctypes.windll.user32
        w = user32.GetSystemMetrics(0)
        h = user32.GetSystemMetrics(1)
        return f'{w}x{h}'

    elif system == 'Darwin':
        result = subprocess.run(
            ['system_profiler', 'SPDisplaysDataType'],
            capture_output=True, text=True
        )
        # парсим строку Resolution
        ...
```

### 12.5 Где фиксируется информация о среде

| Артефакт | Что добавляется |
|---|---|
| `results.json` | поле `environment` (полный блок, см. 12.1) |
| `report.html` | в мета-блоке: ОС, пакет, разрешение экрана, версия редактора |

### 12.6 Платформо-зависимый код — архитектура

Логика теста и платформо-зависимый код **разделены**:

```
Testing/
  AGENTS.md              ← общие правила (корень проекта)
  SCRIPT_RULES.md        ← правила скриптов (корень проекта)
  REPORT_STYLE_RULES.md  ← стандарт отчётов (корень проекта)
  setup_env.py           ← подготовка окружения
  requirements*.txt      ← зависимости

  DECISION_ENGINE.md     ← движок решений (общий)

  products/
    Editors/             ← продукт: Editors
      RISK_MODEL.md      ← модель рисков Editors
      DECISION_ENGINE.md ← движок решений Editors
      scenarios/smoke/   ← тест-сценарии
      test_data/         ← эталонные файлы
      artifacts/         ← результаты прогонов

    <Продукт2>/          ← другие продукты по аналогии
      RISK_MODEL.md
      ...
```

Тест **не знает**, на какой ОС он работает. Он вызывает:
```python
driver.click_button("OK")       # а не user32.mouse_event(...)
driver.take_screenshot(path)    # а не CopyFromScreen(...)
env = detect_environment()      # автоматически
```

Платформа определяется автоматически при запуске, нужный адаптер подключается без участия пользователя.

### 12.7 Имя папки прогона — с учётом платформы

Формат имени папки прогона включает ОС:

```
products/<Продукт>/artifacts/
  case1_windows-11-x64_20260404_143000/
  case1_astra-1.7_20260404_150000/
  case1_macos-arm64_20260404_160000/
  multi_windows-11-x64_20260404_170000/
```

Это позволяет хранить результаты по всем платформам в одном месте и сравнивать.

### 12.8 Окружение и зависимости

#### Почему виртуальное окружение обязательно

- На Linux-дистрибутивах (Astra, ALT, RedOS) системный Python защищён — установка пакетов через `pip` в системный Python запрещена или требует `--break-system-packages`.
- На разных стендах могут быть разные версии Python. Окружение фиксирует зависимости.
- Агент (CI, ручной запуск, cron) должен работать изолированно, не ломая и не завися от системы.

#### Структура окружения

#### setup_env.py — логика

```python
import platform, subprocess, sys, venv
from pathlib import Path

VENV_DIR = Path(__file__).parent / '.venv'

def create_venv():
    if not VENV_DIR.exists():
        venv.create(str(VENV_DIR), with_pip=True)

def pip_path():
    if platform.system() == 'Windows':
        return str(VENV_DIR / 'Scripts' / 'pip')
    return str(VENV_DIR / 'bin' / 'pip')

def install_deps():
    pip = pip_path()
    # общие зависимости
    subprocess.run([pip, 'install', '-r', 'requirements.txt'], check=True)

    # платформо-специфичные
    system = platform.system()
    req_file = {
        'Windows': 'requirements-windows.txt',
        'Linux':   'requirements-linux.txt',
        'Darwin':  'requirements-macos.txt',
    }.get(system)

    if req_file and Path(req_file).exists():
        subprocess.run([pip, 'install', '-r', req_file], check=True)

if __name__ == '__main__':
    create_venv()
    install_deps()
    print(f'Окружение готово: {VENV_DIR}')
```

## 13. Архитектура слоев

Проект автотестов строится на пяти обязательных слоях. Каждый слой имеет чёткую зону ответственности. Смешение слоёв **запрещено**.

### 13.1 Определение слоёв

| # | Слой | Зона ответственности | Примеры |
|---|------|----------------------|---------|
| 1 | **Test Case Layer** | Описание сценария: последовательность шагов, ожидаемые результаты | `1_Запуск_редактора.py` — секция `main()`, вызовы `open_editor()`, `assert_window_visible()` |
| 2 | **Semantic Actions Layer** | Продуктовые действия: «открыть документ», «сохранить файл», «закрыть предупреждение» | `open_document(path)`, `save_file()`, `dismiss_warning()` |
| 3 | **Semantic Assertions Layer** | Продуктовые проверки: «документ открыт», «вкладка активна», «текст введён» | `assert_document_opened(name)`, `assert_tab_active("Вставка")` |
| 4 | **Platform Driver Layer** | Платформо-зависимое взаимодействие с ОС и UI-фреймворком | `driver.click_button("OK")`, `driver.find_element_by_name("Файл")`, `driver.take_screenshot(path)` |
| 5 | **Infrastructure Layer** | Общая инфраструктура: отчёты, скриншоты, среда, таймауты, retry, логирование | `collect_environment()`, `generate_html()`, `build_release_decision()`, `wait_until()` |

### 13.2 Обязательные ограничения

- **Test Case Layer** описывает **только** сценарий. Он вызывает semantic actions и semantic assertions. Он **не содержит** прямого платформенного кода.
- Вызовы `subprocess`, `ctypes`, `xdotool`, `osascript`, `pywinauto`, `pyatspi`, `atomacos`, прямой OCR или Accessibility API **запрещены** в Test Case Layer.
- Платформенные различия **инкапсулированы** в Platform Driver Layer. Тест не знает, на какой ОС работает.
- Semantic Actions вызывают Platform Driver — но **не** Infrastructure, не генерацию отчётов, не принятие решений.
- Semantic Assertions вызывают Platform Driver для получения фактического состояния — но **не** выполняют действий с UI.

### 13.3 Проверяемый критерий

Тест-кейс считается нарушающим архитектуру слоёв, если в файле кейса (`test_cases/`) обнаружен прямой импорт или вызов:

```
subprocess.run, subprocess.Popen, ctypes.windll, ctypes.cdll,
os.system, xdotool, osascript, pywinauto.*, pyatspi.*, atomacos.*,
UIAutomationClient, AT-SPI, AXUIElement
```

Такой кейс **не допускается** к merge.

## 14. Запрет дублирования shared logic

### 14.1 Правило

Если функция или блок логики используется (или потенциально будет использоваться) более чем в одном тест-кейсе, он **обязан** находиться в shared/reusable layer.

### 14.2 Запрет копирования

Копирование общих функций между файлами тест-кейсов **запрещено**. Примеры того, что **не должно** дублироваться:

- сбор информации о среде (`detect_os_info`, `collect_environment`)

- формирование решения о релизе (`build_release_decision`)

### 14.3 Правило второго использования

Если одна и та же логика появилась во втором кейсе — она **немедленно** выносится в общий модуль. Откладывание рефакторинга на потом **запрещено**.

## 15. Контракт `step_result`

### 15.1 Обязательные поля (всегда)

Каждый шаг записывается как `dict` с минимальным набором обязательных полей:

| Поле | Тип | Описание |
|------|-----|----------|
| `step_id` | `str` | Уникальный идентификатор шага в прогоне (например, `"case1_step3"`) |
| `step_name` | `str` | Человекочитаемое название шага на русском языке |
| `status` | `str` | `PASS` \| `FAIL` \| `BLOCKED` |
| `expected` | `str` | Ожидаемый результат |
| `actual` | `str` | Фактический результат (пользовательский, без технических деталей) |
| `screenshot` | `str \| null` | Путь к скриншоту (обязателен для ключевых шагов и всех FAIL) |
| `timestamp` | `str` | ISO 8601 время начала шага |
| `duration_ms` | `int` | Длительность выполнения шага в миллисекундах |
| `critical_path` | `bool` | Принадлежит ли шаг критическому пути |
| `warnings` | `list[dict]` | Непрерывающие предупреждения шага (может быть пустым) |
| `fallback_source` | `str \| null` | Источник fallback (если action выполнен обходным способом) |
| `fallback_reason` | `str \| null` | Причина перехода на fallback |

### 15.2 Обязательные поля при status != PASS

#### Общие поля (обязательны для всех типов сбоев)

| Поле | Тип | Описание |
|------|-----|----------|
| `failure_type` | `str` | `TEST_FAIL` \| `INFRA_FAIL` \| `BLOCKED` (см. раздел 20) |
| `failure_detail` | `str` | Подробное описание сбоя / причины блокировки на русском языке |

#### Дополнительные поля по типу сбоя

| Поле | `TEST_FAIL` | `INFRA_FAIL` | `BLOCKED` |
|------|-------------|--------------|-----------|
| `failure_severity` | **обязательно**: `CRITICAL` \| `HIGH` \| `MEDIUM` \| `LOW` | **обязательно** (по умолчанию `MEDIUM`) | **`null`** (не применимо) |
| `failure_area` | **обязательно**: `DATA_INTEGRITY` \| `CORE_FUNCTION` \| `UI_LAYOUT` \| `PERFORMANCE` \| `PLATFORM_SPECIFIC` | **обязательно**: `INFRASTRUCTURE` | **`null`** (не применимо) |

Для `BLOCKED` поля `failure_severity` и `failure_area` **должны** присутствовать в `step_result` со значением `null`. Это позволяет валидатору проверять контракт единообразно: поля всегда есть, но для `BLOCKED` допускается только `null`.

### 15.3 Пример

```python
step_result = {
    "step_id": "case1_step1",
    "step_name": "Запуск редактора",
    "status": "PASS",
    "expected": "Редактор запущен, появляется предупреждение о регистрации",
    "actual": "Редактор запущен, отображается предупреждение",
    "screenshot": "01_warning_visible.png",
    "timestamp": "2026-04-05T14:30:05",
    "duration_ms": 3200,
    "critical_path": True,
    "warnings": [],
    "fallback_source": None,
    "fallback_reason": None,
}
```

### 15.4 Проверяемый критерий

Шаг, записанный без любого из обязательных полей, считается невалидным. Скрипт **обязан** валидировать контракт перед записью шага в результаты. Используется общий валидатор из Infrastructure Layer.

## 16. Разделение action и assertion

### 16.1 Определения

- **Action** — выполняет действие с UI или системой: клик, ввод, навигация, открытие файла.
- **Assertion** — проверяет результат: элемент видим, текст присутствует, вкладка активна.

### 16.2 Обязательные правила

- **Запрещено** выставлять `status = "PASS"` внутри action-функции. Action выполняет действие и **ничего не утверждает** о результате.
- `status = "PASS"` допускается **только** после выполнения assertion, подтвердившего ожидаемый результат.
- Каждый action в сценарии **обязан** сопровождаться assertion, проверяющим его эффект.
- Assertion **не должен** выполнять побочных действий (кликов, ввода, навигации).
- Если тест-кейс требует «нажатия кнопки интерфейса», action обязан пытаться нажать именно кнопку интерфейса; hotkey в таком шаге разрешён только как fallback с явной причиной.

### 16.3 Именование

- Action-функции: глагол действия — `open_*`, `close_*`, `click_*`, `type_*`, `save_*`, `dismiss_*`
- Assertion-функции: `assert_*` — `assert_window_visible()`, `assert_tab_active("Вставка")`, `assert_document_saved()`

### 16.4 Пример правильного сценария

```python
# Action — выполняет действие
open_editor(editor_path)

# Assertion — подтверждает результат → только здесь ставится PASS/FAIL
result = assert_editor_running(pid)
record_step(step_id="case1_step1", status=result.status, ...)
```

### 16.5 Пример нарушения

```python
# НАРУШЕНИЕ: action сам ставит PASS
def open_editor(path):
    subprocess.Popen([path])
    time.sleep(3)
    return {"status": "PASS", ...}  # ← запрещено
```

## 18. Централизованные wait / retry / timeout

### 18.1 Запрет прямого `time.sleep()` в Test Case Layer

Прямое использование `time.sleep()` в сценарной логике тест-кейса **запрещено**.

### 18.2 Допустимое использование sleep

`time.sleep()` допустим **только** внутри Infrastructure Layer или Platform Driver Layer и **только** с комментарием, объясняющим необходимость:

```python
# Infrastructure Layer — waits.py
def wait_until(condition, timeout_sec=10, poll_interval=0.3):
    """Ожидание выполнения условия с polling."""
    deadline = time.time() + timeout_sec
    while time.time() < deadline:
        if condition():
            return True
        time.sleep(poll_interval)  # polling interval внутри инфраструктурного метода
    return False
```

### 18.3 Обязательные правила

- Все ожидания в тест-кейсах выполняются через централизованные `wait_*` функции из Infrastructure Layer.
- `retry` реализуется централизованно (функция `retry(action, max_attempts, delay)`).
- `timeout` **параметризуемый**: каждая `wait_*` функция принимает `timeout_sec` с разумным значением по умолчанию.
- Значения timeout по умолчанию определяются в конфигурации, а не хардкодятся в каждом кейсе.

### 18.4 Стандартные wait-функции (Infrastructure Layer)

| Функция | Назначение |
|---------|------------|
| `wait_until(condition, timeout_sec)` | Ожидание произвольного условия |
| `wait_window_appear(title, timeout_sec)` | Ожидание появления окна |
| `wait_element_enabled(name, timeout_sec)` | Ожидание доступности элемента |
| `wait_process_start(name, timeout_sec)` | Ожидание запуска процесса |
| `retry(action, max_attempts, delay)` | Повтор действия при неуспехе |

### 18.5 Проверяемый критерий

`grep -n "time.sleep" test_cases/` — если найдены прямые вызовы `time.sleep()` в файлах тест-кейсов, это нарушение. Такой код **не допускается** к merge.

## 19. Политика fallback

### 19.1 Порядок fallback (приоритет способа взаимодействия)

При взаимодействии с UI элементами соблюдается строгий порядок:

| Приоритет | Способ | Когда используется |
|-----------|--------|-------------------|
| 1 (основной) | **Accessibility API** (UIA / AT-SPI / AXUIElement) | Всегда по умолчанию |
| 2 | **OCR / text-based detection** | Если Accessibility API недоступен или не находит элемент |
| 3 | **Координатный fallback** | Только если OCR не помог; обязательно помечается как fallback |
| 4 (крайний) | **CV (computer vision)** | Последний резерв; только если все предыдущие не работают |

### 19.2 Обязательное логирование fallback

- Каждый факт перехода на fallback-способ **обязан** логироваться.
- В логе фиксируется: какой основной способ не сработал, какой fallback использован, причина перехода.
- Факт fallback **обязан** попадать в артефакты прогона (`results.json`, `report.html`, `report.md`) как часть шага, а не только в runtime-лог.

```python
# Пример логирования fallback
log.warning(
    "FALLBACK: Accessibility API не нашёл кнопку 'OK' (UIA timeout 5s). "
    "Переход на OCR-поиск текста 'OK'."
)
```

### 19.3 Координатный fallback

Использование координатного взаимодействия (клик по абсолютным координатам) **обязано** быть явно помечено:

```python
# FALLBACK: координатный клик — Accessibility и OCR не нашли кнопку OK
# Причина: диалог регистрации не экспонирует кнопку через UIA
driver.click_at(x=540, y=380)
```

### 19.4 Проверяемый критерий

Координатные действия без пометки `FALLBACK` и без указания причины в комментарии **не допускаются** к merge.

### 19.5 Вкладки ленты (toolbar tabs): устойчивость к неполной калибровке

Для вкладок ленты, где координаты получаются через OCR-калибровку
(`calibrate_toolbar_tabs`), обязательны следующие правила:

- Первичная калибровка может быть неполной на некоторых DPI/масштабах и
  локалях (например, одна вкладка не распознана OCR).
- Action `click_toolbar_tab` обязан иметь fallback "on-demand OCR click":
  если координаты конкретной вкладки отсутствуют, перед FAIL выполняется
  свежий скриншот и OCR-поиск имени этой вкладки с кликом по найденной метке.
- Ошибка "координаты вкладки не откалиброваны" допускается только после
  исчерпания fallback для этой вкладки.
- Для многословных названий вкладок допускается искать устойчивый токен
  (например, первое слово), если это повышает стабильность OCR.

### 19.6 Debug-launch fallback цепочка (обязательно)

- Для Windows-launch использовать двухшаговый запуск:
1. основной путь: запуск с debug-флагом (`--ascdesktop-support-debug-info`);
2. fallback: при отсутствии главного окна в timeout — перезапуск без debug-флага.
- Если debug-порт/CDP недоступен, action-слой обязан перейти на fallback-цепочку (UIA/OCR/coordinates/CV) и зафиксировать это в шаге.
- Если оба launch-режима не дали главное окно, шаг классифицируется как `INFRA_FAIL` или кейс как `BLOCKED` (по контексту предусловия), с явной причиной.

### 19.7 DOM-источник для Editors

- Для Editors допускается DOM/CDP как primary для CEF-элементов (стартовое меню, вкладки ленты, редакторный фрейм).
- Исключение: область страницы документа (`canvas` во `viewport`) не содержит стабильной DOM-модели текста документа.
- Для шагов «ввод/редактирование текста на странице документа» DOM/CDP не используется как источник верификации текста.
- Для таких шагов фокус через viewport (координатный клик в рабочую область) считается штатным primary-путём, а не fallback.
- Предупреждения о недоступности DOM в canvas-области документа не выводятся в отчёт.
- HTML-снимки из `products/Editors/html/` используются как эталон селекторов и структуры.
- HTML-снимки являются reference-артефактами: редактирование файлов в `products/Editors/html/*` запрещено в рамках обычного тестового рефакторинга.
- Для выбора эталона по типу сценария использовать явное соответствие:
  - сценарий для документа (`.docx`) → `products/Editors/html/Документ1_docx.html`
  - сценарий для таблицы (`.xlsx`) → `products/Editors/html/Книга1_xlsx.html`
  - сценарий для презентации (`.pptx`) → `products/Editors/html/Презентация1_pptx.html`
  - сценарий с главным окном редактора (shell/start/menu-level) → `products/Editors/html/Главное_окно.html`

### 19.8 UI-каталог секций/вкладок

- Продуктовые проверки UI должны переиспользовать централизованный каталог (например, `products/Editors/assertions/ui_catalog.py`), а не дублировать списки по кейсам.
- Новые элементы UI, отсутствующие в каталоге, фиксируются как warning (`UI_NEW_ELEMENT`) и не переводят шаг в FAIL.
- Отсутствие ожидаемого элемента из каталога фиксируется как warning (`UI_MISSING_ELEMENT`) или `TEST_FAIL` — по правилу конкретного assertion.

## 20. Классификация типов сбоев

### 20.1 Типы сбоев

Каждый сбой классифицируется по **типу** (причина) и **серьёзности** (влияние). Тип сбоя записывается в поле `failure_type` контракта `step_result`.

| Тип | Код | Описание | Влияние на решение |
|-----|-----|----------|-------------------|
| Продуктовый сбой | `TEST_FAIL` | Проблема в тестируемом продукте: функция не работает, крэш, потеря данных | Определяется по `failure_severity` через `DECISION_ENGINE.md` |
| Инфраструктурный сбой | `INFRA_FAIL` | Проблема среды, инструмента или драйвера: Tesseract не установлен, UIA timeout, скриншот не сохранился, нет доступа к файлу | Не является сигналом о качестве продукта. Требует повторного прогона |
| Заблокирован | `BLOCKED` | Кейс невозможно исполнить по внешней причине: редактор не установлен, лицензия истекла, тестовый файл отсутствует, предыдущий кейс цепочки не прошёл | Кейс не даёт сигнала. Покрытие снижено |

### 20.2 Обязательные правила классификации

- `TEST_FAIL` — ставится **только** когда продукт демонстрирует дефект.
- `INFRA_FAIL` — ставится, когда тест не смог выполниться из-за проблем среды/инструмента, **а не** из-за проблем продукта.
- `BLOCKED` — ставится, когда выполнение кейса невозможно по внешней причине, не связанной ни с продуктом, ни с инфраструктурой тестирования.
- **Запрещено** маркировать инфраструктурный сбой как `TEST_FAIL`. Это искажает оценку качества продукта.
- **Запрещено** маркировать продуктовый дефект как `INFRA_FAIL`. Это скрывает реальные проблемы.
- Сбой вида "координаты элемента/вкладки не откалиброваны" классифицируется
  как `INFRA_FAIL`, если основной путь и fallback (OCR/другие допустимые
  способы) исчерпаны и assertion продукта не выполнялся.

### 20.3 Как тип сбоя отражается в результатах

| Поле в step_result | TEST_FAIL | INFRA_FAIL | BLOCKED |
|---------------------|-----------|------------|---------|
| `status` | `FAIL` | `FAIL` | `BLOCKED` |
| `failure_type` | `TEST_FAIL` | `INFRA_FAIL` | `BLOCKED` |
| `failure_severity` | по классификации | `MEDIUM` (по умолчанию) | `null` |
| `failure_area` | по классификации | `INFRASTRUCTURE` | `null` |
| `failure_detail` | описание дефекта продукта | описание проблемы среды | причина блокировки |

### 20.4 Влияние на итоговый прогон

- `TEST_FAIL` — учитывается в `DECISION_ENGINE.md` при формировании вердикта.
- `INFRA_FAIL` — **не учитывается** как продуктовый сигнал. Вместо этого снижает достоверность прогона (см. `DECISION_ENGINE.md`, раздел «Достоверность прогона»).
- `BLOCKED` — **не учитывается** как продуктовый сигнал. Снижает покрытие критических путей.

### 20.5 Обязательные счётчики в summary

```python
"summary": {
    "total": 10,
    "passed": 7,
    "failed": 1,       # только TEST_FAIL
    "infra_failed": 1, # только INFRA_FAIL
    "blocked": 1,       # только BLOCKED
}
```

## 21. Product-specific layer

### 21.1 Разделение общей инфраструктуры и продуктовой логики

- **Общая инфраструктура** (shared/) — код, не зависящий от продукта: отчёты, скриншоты, среда, waits, retry, decision engine.
- **Продуктовый слой** (Editors/actions/, Editors/assertions/) — semantic actions и assertions, специфичные для конкретного продукта.

### 21.2 Обязательные правила

- **Запрещено** смешивать общую инфраструктуру и продуктовую бизнес-логику в одном модуле.
- Общий модуль **не должен** содержать импортов или упоминаний конкретного продукта (Editors, r7-office и т.д.).
- Продуктовый модуль **может** импортировать из shared/, но **не** наоборот.
- При добавлении нового продукта (например, `Spreadsheets/`) его semantic actions/assertions создаются **отдельно**, не в shared/.

### 21.3 Проверяемый критерий

Если в `shared/` обнаружен импорт из `Editors/` или другого продукта — это нарушение. Такой код **не допускается** к merge.

## 22. Контрольный раздел: проверка перед merge

### 22.1 Недопустимые нарушения (блокируют merge)

Следующие нарушения считаются **блокирующими** — код не принимается в основную ветку:

| # | Нарушение | Как проверить |
|---|-----------|--------------|
| 1 | Прямой платформенный код в Test Case Layer | `grep` по `subprocess`, `ctypes`, `xdotool`, `osascript` в `test_cases/` |
| 2 | Дублирование shared logic между кейсами | Одна функция определена в двух файлах `test_cases/` |
| 3 | `time.sleep()` в Test Case Layer | `grep "time.sleep" test_cases/` |
| 4 | Action выставляет PASS без assertion | Code review: action-функция возвращает `status: PASS` |
| 5 | Отсутствие обязательных полей в step_result | Валидатор контракта при запуске скрипта |
| 6 | `TEST_FAIL` при инфраструктурном сбое | Code review: тип сбоя не соответствует причине |
| 7 | Координатный клик без пометки FALLBACK | `grep` по координатным вызовам без комментария `FALLBACK` |
| 8 | Импорт из продуктового слоя в shared/ | `grep` по `from Editors` / `import Editors` в `shared/` |
| 9 | Нарушение конвенции именования | Code review: action/assertion функция не соответствует §17 |
| 10 | Отсутствие блока `release_decision` в результатах | Валидатор при генерации отчёта |

### 22.2 Признаки неправильно написанного теста

Новый тест считается написанным **неправильно**, если:

- содержит `>50 строк` кода, не относящегося к сценарию (инфраструктура, отчёты, среда);
- импортирует `subprocess`, `ctypes` или платформенные библиотеки напрямую;
- определяет функции, уже существующие в shared/ или actions/;
- не использует централизованные `wait_*` для ожиданий;
- не классифицирует сбои по типу (`TEST_FAIL` / `INFRA_FAIL` / `BLOCKED`);
- не содержит `CASE_META` с привязкой к модели рисков;
- не формирует блок `release_decision`.

## 23. Целевая модель верификации (обязательная привязка)

Для этапов миграции и новых сценариев Editors использовать
`docs/methodology/TARGET_MODEL_GLOSSARY.md` как канонический источник:

- единый glossary (`UI-state-driven`, `file-model-backed`, `geometry-aware`,
  feature-based visual verification, risk-based decisioning);

Проверяемый критерий:
- при ревью сценария должно быть явно видно, какие approved sources использованы
  для каждого критического шага;
- если источник выбран вне approved-списка, это считается отклонением и требует
  отдельного согласования с пользователем.
