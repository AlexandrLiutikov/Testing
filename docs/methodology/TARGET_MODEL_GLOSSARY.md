# Target Model Glossary (Editors)

Документ фиксирует целевую терминологию и критерии миграции для подхода
`UI-state-driven` + `file-model-backed` + `geometry-aware` + feature-based visual verification + risk-based decisioning.

---

## 1. Единый glossary

### `UI-state-driven`
- Автотест принимает решения на основании наблюдаемого состояния UI: доступность, активность, видимость, выбранная вкладка, текстовые маркеры.
- PASS допустим только после подтверждающего assertion состояния интерфейса.

### `file-model-backed`
- Проверка пользовательского действия подтверждается не только UI-сигналом, но и файловой моделью результата: открытие, сохранение, изменение содержимого, целостность формата.
- Для сценариев сохранения/открытия артефакт файла является обязательным источником верификации.

### `geometry-aware`
- Геометрические признаки (позиция, размер, перекрытие, смещение) используются как проверяемый сигнал, но с учетом tolerance из `products/Editors/RISK_MODEL.md`.
- Геометрия не заменяет semantic assertions; она усиливает диагностику визуальных отклонений.

### `feature-based visual verification`
- Визуальная проверка выполняется по признакам продукта (feature-level), а не по «сырым» пикселям.
- Примеры признаков: активная вкладка, наличие секции ribbon, доступность кнопки, наличие обязательных UI-элементов каталога.

### `risk-based decisioning`
- Итоговый вердикт строится через модель рисков продукта и `DECISION_ENGINE.md`.
- В решение попадают severity/area, покрытие critical path, типы `TEST_FAIL`/`INFRA_FAIL`/`BLOCKED` и `run_confidence`.

---

## 2. Approved Verification Sources

Ниже — утвержденные источники верификации для Editors (в порядке приоритета):

1. Semantic assertions на Accessibility API (UIA / AT-SPI / AXUIElement).
2. DOM/CDP assertions для CEF-частей Editors (по правилам `SCRIPT_RULES.md`, раздел 19.7).
3. File model checks (существование файла, формат, целостность, ожидаемое содержимое/структура).
4. OCR text verification (устойчивые режимы и мягкое совпадение).
5. Geometry signals (смещение, размеры, перекрытия) с учетом tolerance из risk model.
6. Координатный/CV fallback только как резервный источник после исчерпания основных.

Обязательные ограничения:
- Запрещено опираться на единственный низкоприоритетный источник, если доступен более приоритетный.
- Любой fallback обязан быть явно отражен в `fallback_source`, `fallback_reason`, `warnings`.
- Для critical path проверка должна опираться минимум на один semantic source (пп. 1-3).

---

## 3. Definition of Done для миграции на целевую модель

Миграция сценария/набора сценариев считается завершенной только если выполнено все:

1. Для каждого шага определен основной verification source из approved-списка.
2. Для каждого action предусмотрен assertion, подтверждающий пользовательский эффект.
3. PASS не выставляется внутри action; PASS ставится только после assertion.
4. В сценариях сохранения/открытия добавлена `file-model-backed` проверка результата.
5. Геометрические проверки используют tolerance из `RISK_MODEL.md` и не подменяют semantic-проверку.
6. Все fallback-переходы зафиксированы в результатах шага.
7. Все non-PASS классифицированы строго как `TEST_FAIL` / `INFRA_FAIL` / `BLOCKED`.
8. `release_decision` и `run_confidence` формируются по `DECISION_ENGINE.md`.
9. Нет дублирования reusable-логики; действия/проверки находятся в корректных слоях (`shared/`, `actions/`, `assertions/`).
10. Документация синхронизирована: ссылки на target model добавлены в канонические правила продукта и методологии.
